# PR 생성 프로세스

## 생성 워크플로우

> **절대 규칙**: 아래 단계를 순서대로 수행할 것. 단계를 건너뛰거나 한 번에 처리 금지.

### 0단계: 사전 검증

PR 생성 전 반드시 아래 항목을 확인:

- **GitHub CLI 인증**: `gh auth status`로 로그인 상태 확인. 미인증 시 `gh auth login`을 안내하고 중단
- **원격 저장소 연결**: `git remote -v`로 원격 저장소가 올바르게 설정되어 있는지 확인
- **민감 파일 제외**: `git status`에서 `.env`, 시크릿, 인증 파일이 스테이징에 포함되지 않았는지 확인. 발견 시 경고 후 중단
- **브랜치 확인**: 현재 브랜치가 PR 소스 브랜치(보통 `dev`)인지 확인

### 1단계: 상태 분석

- `git status`로 미커밋 변경사항 확인
- `git log --oneline <target>..<source>`로 포함될 커밋 목록 확인
- `git diff --stat <target>..<source>`로 변경 파일 통계 확인
- 미커밋 변경사항이 있으면 **사용자에게 커밋 여부를 확인**한 후 처리

### 2단계: 원격 푸시

- `git push origin <branch>`로 최신 커밋을 원격에 반영
- 푸시 실패 시 원인을 분석하고 사용자에게 보고 (force push는 사용자 승인 없이 금지)

### 3단계: PR 초안 작성

커밋 분석 결과를 바탕으로 초안을 작성하되, 반드시 아래 파일을 참조:

- **템플릿**: `.github/PULL_REQUEST_TEMPLATE.md`의 구조를 그대로 따를 것
- **릴리스 라벨**: `.github/release.yml`의 카테고리와 라벨 매핑을 참조하여 변경 유형 판단

> 템플릿 파일이 변경되었을 수 있으므로, PR 생성 시마다 **최신 파일을 읽어서 확인**할 것.

작성 항목:

- **요약**: 전체 변경의 목적을 한두 문장으로 요약
- **변경 내용**: 커밋을 유형별(기능, 수정, 문서 등)로 그룹화하여 정리
- **변경 유형**: 커밋 프리픽스 기반으로 해당 항목 체크
- **테스트**: 수행한 테스트 항목 체크
- **참고 사항**: 커밋 수, 파일 변경 통계, 리뷰어가 알아야 할 정보

#### PR 제목 규칙

- Conventional Commits 형식 사용: `<type>: <설명>`
- 제목에 **버전 번호를 포함하지 않음** (버전은 릴리스 태그에서 관리)
- 70자 이내로 작성
- 한국어 또는 영어 (프로젝트 컨벤션에 따름)
- 예: `chore: 개발 환경 구성 및 프로젝트 설정`, `feat: 파일 업로드 기능 추가`

#### 본문 작성 규칙

- 모든 내용은 **한국어**로 작성
- 변경 내용은 구체적이고 명확하게 기술
- 커밋 메시지를 그대로 복사하지 말고, 의미 단위로 재구성

### 4단계: 사용자 검토

- 제목 + 본문 초안을 사용자에게 보여주고 **명시적 승인 후** 다음 단계로 진행
- 사용자가 수정을 요청하면 반영 후 재검토

### 5단계: PR 생성 및 라벨 부여

```bash
gh pr create --base <target> --head <source> --title "<제목>" --body "<본문>"
gh pr edit <PR번호> --add-label "<라벨1>,<라벨2>"
```

- 기본 타깃: `master` (브랜치 전략: `dev → PR → master`)
- PR 생성 후 반드시 라벨을 추가 (라벨 규칙 참조)
- 생성된 PR URL을 사용자에게 보고

## 머지 방식

- **Squash merge** 사용 권장
- master 히스토리를 PR 단위로 깔끔하게 유지
- 릴리스 노트 자동 생성과 호환성 유지

## 라벨 규칙

### 커밋 프리픽스 → 라벨 매핑

PR에 포함된 커밋의 프리픽스를 분석하여 해당하는 라벨을 모두 부여:

| 커밋 프리픽스 | GitHub 라벨 | 릴리스 노트 카테고리 |
|--------------|-------------|-------------------|
| `feat:` | `enhancement` | 🚀 새 기능 |
| `fix:` | `bug` | 🐛 버그 수정 |
| `chore:` | `chore` | 🔧 설정 및 환경 |
| `docs:` | `documentation` | 📝 문서 |
| `refactor:` | `refactor` | ♻️ 리팩토링 |
| `test:` | `test` | 🧪 테스트 |

> `.github/release.yml`의 카테고리 설정과 연동되므로, 라벨명을 임의로 변경하지 말 것.

### 라벨 생성

레포에 필요한 라벨이 없을 경우 아래 명령으로 생성:

```bash
gh label create <이름> --description "<설명>" --color "<색상코드>"
```

기본 라벨 목록:

- `enhancement` — 새 기능 (#a2eeef)
- `bug` — 버그 수정 (#d73a4a)
- `chore` — 설정/환경 (#ededed)
- `documentation` — 문서 (#0075ca)
- `refactor` — 리팩토링 (#d4c5f9)
- `test` — 테스트 (#bfd4f2)
- `infrastructure` — 인프라 (#f9d0c4)

## 주의사항

1. **gh CLI 미인증 시 PR 생성을 시도하지 말 것** — 반드시 0단계에서 인증 확인
2. **force push 금지** — 푸시 충돌 시 사용자에게 상황을 설명하고 해결 방안 제시
3. **PR 제목에 버전 번호 금지** — 버전은 릴리스 태그에서 관리, 제목과 태그 불일치 방지
4. **민감 파일 체크 필수** — `.env`, 인증 토큰, 시크릿 파일이 커밋에 포함되지 않았는지 확인
5. **사용자 승인 없이 PR 생성 금지** — 반드시 초안 검토 후 승인을 받을 것
6. **이 규칙은 PR 프로세스에만 적용** — 릴리스, 배포 등 다른 프로세스와 독립적으로 관리
7. **템플릿 준수 필수** — PR 본문은 반드시 `.github/PULL_REQUEST_TEMPLATE.md`를 읽고 그 구조를 따를 것. 템플릿 없이 임의 형식으로 작성 금지
